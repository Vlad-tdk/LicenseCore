🧱 Project Architecture

The system is built on three key components, each solving its own task. Plus there are specialized variants for different use cases.

1. 🧠 License Generation (HMAC-SHA256)

Why HMAC instead of RSA or ECC?

Initially, I wanted to use RSA — asymmetric cryptography seemed more reliable. But I quickly realized the problems:

  • RSA signatures take up a lot of space (256+ bytes)
  • Require heavy computational resources in browsers
  • Public key still needs to be stored somewhere in the client

HMAC-SHA256 solves these problems elegantly:

  • Signature is only 64 hex characters (32 bytes)
  • Lightning-fast verification even on weak hardware
  • Secret key never leaves the server

How hardware binding works:

// Get unique hardware fingerprint std::string hwid = hardware_fingerprint.get_fingerprint(); // Combines: CPU ID + MAC address + disk serial // Create data for signing std::string license_data = user_id + expiry + features + hwid; // Sign with HMAC-SHA256 std::string signature = hmac_validator.sign(license_data);

Server process (CLI or API):

# License generation via CLI ./license_gen --user "user-123" --expire "2025-12-31" --hwid "a1b2c3d4..." --features "premium,api" # Result: signed JSON { "user_id": "user-123", "hardware_hash": "a1b2c3d4e5f6...", "expiry": "2025-12-31T23:59:59Z", "features": ["premium", "api"], "hmac_signature": "c4ef45e6a7b8..." }

2. 🌐 Client-side License Validation

WASM Library in Browser:

The main feature of the project — the same validation logic works everywhere. C++ code compiles to WebAssembly and embeds directly in the browser:

// Initialize WASM module const licenseCore = await loadLicenseCore(); const manager = new licenseCore.LicenseManager("secret-key"); // License validation try { const licenseInfo = manager.loadAndValidate(licenseJson); if (licenseInfo.valid && manager.hasFeature("premium")) { // Enable premium features enablePremiumFeatures(); } } catch (error) { showLicenseError(error.message); }

Why validation in JS?

  • WebUI activation: License keys directly in browser
  • Embedded frontend: Built-in web interfaces in desktop applications
  • Demo versions: Feature limitation by license
  • Offline applications: PWA with local validation

🛡️ Reverse Engineering Protection

Yes, JavaScript code can be inspected. But:

  1. WASM obfuscation: C++ sources hidden in binary format
  2. Runtime checks: HMAC key not stored in code, passed during initialization
  3. Additional layers: anti-debug, DevTools detection
  4. Server validation: Critical operations duplicated on server

⚔️ What Breaks Simple Protection — And How I Fixed It

Let me show you the common vulnerabilities in basic licensing systems and how LicenseCore++ addresses them:

Problem #1: Embedded Keys

// ❌ Bad: Secret directly in code bool check_license(const std::string& key) { return key == "SUPER-SECRET-123"; // visible in disassembler in 30 seconds } // ✅ Good: HMAC with server key bool validate_license(const std::string& license_json) { // Secret key remains on generation server // Client only checks signature return hmac_validator.verify_json(license_json); }

Problem #2: Simple Time Checks

// ❌ Bad: System time check if (std::time(nullptr) > license_expiry) { return false; // easily bypassed by changing date } // ✅ Good: Signed expiry date // Date is embedded in HMAC signature, cannot be changed auto info = parse_and_verify_license(license_json); return info.expiry > std::chrono::system_clock::now();

🔑 Generation vs Validation Logic Separation

Key principle — generation server and validation client have different rights:


Server (has secret key):
├── Generates HMAC signatures
├── Creates licenses
└── Can revoke licenses (CRL)

Client (has only validator):
├── Checks HMAC signatures
├── Reads license data
└── CANNOT create new licenses
